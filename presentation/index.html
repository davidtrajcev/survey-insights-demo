The brief, decoded

The survey is the easy part.

P&C asked for: two cycles a year (annual + half-year), a continuous eNPS pulse, four themes — health, workload, leadership, collaboration — manager reporting down the org tree, a <4 anonymity rule, support for org changes, trends over time, ~70% response, and "simple enough that managers use it."

  • Anonymity that actually holds — the <4 rule alone leaks.
  • Org changes that don't rewrite history — reorgs are constant.
  • Adoption — insight in one glance, or it goes unused.

The requirement that breaks inference leak

"<4 is hidden" + "managers see everyone below" = anonymity by subtraction.

40 − 12 − 15 − 10 = 3  →  the "hidden" team is re-identified.

The threshold protects nothing if the surrounding numbers are visible. Break this once and you lose the trust the whole tool runs on — and it's a GDPR / works-council exposure. Fix: layered suppression — primary (<4), secondary (hide a sibling so the remainder can't be isolated), per-question, and cross-dashboard. I implemented all four.

Design · org structure

Org changes don't rewrite history — because history is frozen at the snapshot.

Each cycle
pins an org snapshot — the structure at that moment. Reorgs create a new snapshot; old ones are never mutated.
Identity
a stable external_key tracks the same business unit across snapshots, so a renamed/moved team keeps one continuous trend line.
Attribution vs access
responses are tagged with the org unit at submission time (frozen); who can view is resolved against the current org.

That split is the whole answer to "people move, managers change, teams restructure": the numbers a manager saw last cycle never silently change, and access still follows today's reporting lines.

Design · anonymisation in practice

A team of 3 inside a department of 40 — what does the manager see?

Customer Operations32visible
↳ Core Support14visible
↳ Customer Success12visible
↳ Leadership—suppressed
↳ CX Research Pod🔒hidden <4

32 − 14 − 12 = 6 = pod + Leadership → still ≥ 4, the pod can't be isolated.

Why Leadership is hidden too

Hiding only the 3-person pod would let a manager recover it by subtraction. So the smallest visible sibling is suppressed with it — the most you can back out is (pod + sibling), a group ≥ 4 that can't be unpicked. Runs to a fixpoint up the tree.

And the corners

Per-question suppression, cross-dashboard masking (drilling in can't reopen the attack), and the monthly eNPS pulse masked the same way. Counts are never shown for hidden groups.

Design · storage & the privacy boundary

Identity and answers are stored on opposite sides of a wall.

Participant record (who)

SSO-authenticated employee, eligibility, org snapshot unit, submitted / not-submitted. Used for access, eligibility, and duplicate prevention.

Response record (what)

Anonymous answers only — no employee id, email, name, SSO subject, or participant id, and only a coarse submission month, never a precise timestamp.

Managers read privacy-safe aggregates only, through a layer that applies suppression before anything reaches a template. The raw answer is never joinable back to a person.

Design · trends, not snapshots

Movement is the product — not a single score.

  • Weighted category scores — each theme is a weighted mean of its questions, so the questions that matter most count for more.
  • Movement across cycles — every theme shows its delta vs the previous cycle, with the biggest mover surfaced.
  • Company benchmark — each theme compared to the org-wide average, so a manager knows what's local vs systemic.
  • eNPS as its own cadence — a continuous monthly pulse on a −100…100 scale, trended separately from the half-year themes.

Design · simple enough to use

One screen answers "what do I do with this?"

  • The cockpit — a priority-weighted focus area: the one theme to raise in the next team conversation, framed as a prompt, not a verdict.
  • Role-aware navigation — employee, manager, and P&C admin each land where they can act.
  • Honest about privacy — every dashboard says what's hidden and why, so a manager trusts what they're seeing.

My stance on Q1

Ship the insight layer first with one opinionated action per cycle. Add tracking/action-plans only once managers ask — heavy workflow is what kills adoption. "Simple" is a feature, not a cut corner.

Design · I stress-tested my own privacy security review

Then I tried to break my own anonymity guarantees — and found four leaks.

eNPS bypassed the cross-dashboard mask

The pulse history ignored the selected cycle, so a suppressed team was readable via an older one. Now each pulse is masked as its own publication.

fixed

A concurrency race (3 → 4)

Two submissions could both pass the "already submitted?" check and push a hidden team to the threshold. Closed with an atomic claim on the participant.

fixed

Two hidden teams = a recoverable 3

The "exactly one hidden child" rule missed teams of 1 + 2 summed together. Now I check the combined residual, not the row count.

fixed

A manager's own answer was recoverable

Subtracting the visible teams left a lone manager's single direct response. Direct respondents are now modelled as an implicit child cohort.

fixed

From demo to production

The demo proves the model; production hardens it — and adds the AI layer.

Platform

Today: FastAPI + SQLite, deterministic seed. In production: Entra ID / OIDC SSO; Aurora as the source of truth (a read cache only if traffic needs it); suppression materialised at cycle close; effective-dated org from the HRIS.

Where AI fits

Summarise open-text comments into themes, suggest the focus area, and draft manager talking points — all computed over already-suppressed aggregates, so the privacy boundary holds for the model too.

The rule-based cockpit today is the honest seam where an LLM slots in next — same inputs, richer output.

Production architecture · AWS

Production: serverless compute, behind a CDN.

CloudFront routes static traffic to S3 and API requests to API Gateway; authenticated requests invoke the same FastAPI application on Lambda.
